# Crypto P2P Scanner Bot

Crypto P2P Scanner Bot - це Telegram-бот для швидкого перегляду та фільтрації P2P-ордерів на Binance і OKX.

Проєкт створений для сценаріїв, де важливо не просто побачити курс, а швидко оцінити якість ордера: умови мерчанта, спосіб оплати, рейтинг, кількість угод, час виконання та ризикові формулювання в описі.

## Що вміє бот

- Показує P2P-ордери Binance та OKX.
- Підтримує напрямки UAH -> USDT та USDT -> UAH.
- Виводить ордери з пагінацією прямо в Telegram.
- Зберігає персональні фільтри для кожного користувача.
- Дозволяє налаштовувати кількість ордерів у видачі.
- Кешує запити до бірж, щоб зменшити навантаження та прискорити повторні перегляди.
- Має базову адмін-панель з рольовим доступом.

## Фільтрація P2P-ордерів

Бот допомагає відсіювати ордери, які не підходять за умовами або виглядають ризиково.

Доступні фільтри:

- максимальний час угоди;
- мінімальна кількість угод мерчанта;
- мінімальна оцінка мерчанта;
- мінімальний відсоток виконання;
- типи оплати: фізособа/карта, ФОП/ТОВ/IBAN, інші методи;
- оплата від третіх осіб;
- оплата кількома платежами;
- оплата через Monobank "банку";
- кількість кандидатів, які потрібно перевіряти перед видачею;
- режим перевірки опису: Regex, GPT або Regex + GPT.

## Перевірка описів

Окрема увага в проєкті приділена описам P2P-ордерів. Саме там мерчанти часто вказують важливі умови: оплату частинами, оплату від довірених осіб, перекази на ФОП/ТОВ, Monobank "банку", довге очікування або інші обмеження.

Для цього бот має два підходи:

- **Regex** - швидка перевірка очевидних формулювань.
- **GPT** - гнучкіша перевірка складних або нечітких описів через OpenAI Responses API.

Також доступний комбінований режим **Regex + GPT**, де бот спочатку прибирає очевидні невідповідні ордери, а потім додатково перевіряє решту через GPT.

## Біржі

### Binance

Для Binance бот отримує список P2P-ордерів і додатково підтягує деталі ордера, зокрема опис мерчанта. Це дозволяє якісно застосовувати фільтри по умовах.

### OKX

Для OKX бот працює зі списками P2P-ордерів і намагається підтягувати деталі оголошень. Опис OKX може бути доступний не завжди, оскільки частина detail-запитів залежить від web-контексту біржі. Якщо опис не вдається отримати, такі ордери можна вважати неперевіреними при активних фільтрах по опису.

## Особистий кабінет

У користувача є особистий кабінет, де можна переглянути інформацію про себе та налаштувати P2P-фільтри.

Фільтри зберігаються в базі даних, тому після перезапуску бота користувач не втрачає свої налаштування.

## Ролі та права

У проєкті реалізовано RBAC-модель:

- **User** - базовий користувач бота.
- **Admin** - користувач з доступом до адміністративних функцій.
- **Super Admin** - користувач з повним доступом.

Кнопки та доступні дії в Telegram залежать від ролі користувача. Наприклад, адміністратор може бачити адмін-панель і керувати довідниками, а звичайний користувач бачить тільки доступні йому розділи.

## Адмін-панель

Адмін-панель дозволяє керувати довідковими даними через Telegram. Зараз реалізовано керування валютами з контрольованого списку, щоб адмін не додавав випадкові або некоректні значення вручну.

Цей підхід можна розширювати на біржі, методи оплати та інші довідники.

## База даних

Бот використовує PostgreSQL для зберігання:

- користувачів;
- персональних P2P-фільтрів;
- ролей і прав;
- валют;
- бірж;
- методів оплати;
- P2P-ордерів;
- потенційних арбітражних можливостей;
- історії сканувань і сповіщень.

Структура проєкту розділена на моделі, DTO, репозиторії, сервіси, handlers, filters і seeders, щоб логіка не змішувалась в одному місці.

## Технології

- Python
- aiogram 3
- aiohttp
- SQLAlchemy 2
- asyncpg
- PostgreSQL
- OpenAI Responses API

## Архітектура

Проєкт розділений на кілька основних шарів:

- `handlers` - Telegram-команди, меню та callback-обробники;
- `keyboards` - reply та inline клавіатури;
- `services` - бізнес-логіка, клієнти бірж, фільтрація, форматування;
- `repositories` - робота з базою даних;
- `db/models` - SQLAlchemy-моделі;
- `db/dto` - DTO, константи та типи;
- `filters` - перевірка прав доступу;
- `fsm` - стани Telegram-меню;
- `db/seeders` - базові ролі, права і довідники.

## Безпека

У проєкті передбачено роботу з чутливими ключами та токенами через environment variables. Реальні токени Telegram, OpenAI або OKX не повинні потрапляти в git.

Для захисту від зайвого навантаження на біржі реалізовано кешування та cooldown-и між запитами.

## Статус проєкту

Проєкт активно розвивається. Поточний фокус:

- стабільне отримання P2P-ордерів;
- якісна фільтрація описів;
- зручний Telegram UI;
- розширення адмін-панелі;
- підготовка до повноцінного P2P/арбітражного сканера.
